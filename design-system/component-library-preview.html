Button

Contained · Outlined · Text — actual size

Buttons let operators trigger actions — saving a vehicle, running a report, confirming a change. Emphasis communicates priority: the one most important action on a view is the contained button; everything else steps down.

Use exactly one contained (high-emphasis) button per view. The floating action button (FAB) is excluded — it's a mobile pattern; on desktop the primary action sits in the page toolbar / app bar.

Variants

ContainedThe single primary action on a view (Save, Add, Run).
OutlinedImportant secondary actions, or "Cancel" next to a contained button.
TextLow-emphasis actions — in toolbars, cards, and dialog dismissals.
MapList
Toggle button(s)Switch between exclusive views (Map / List); 2–5 options, not navigation.

Best practices

Do
  • Keep one contained action per view.
  • Use verb-first labels ("Add vehicle", not "New").
  • Pair Cancel (outlined/text) beside the primary.
  • Darken the orange fill for legible white text.
Don't
  • Don't stack multiple contained buttons.
  • Don't use a button for navigation that's really a link.
  • Don't ship white-on-orange for important labels.
  • Don't hard-code colours/sizes — use tokens + these specs.

Props

labelstring—Short, verb-first action label (e.g. "Add vehicle"). Rendered uppercase per MD2.
variant'contained' | 'outlined' | 'text'—Contained = single primary action per view. Outlined = important secondary actions. Text = low-emphasis actions.
leadingIconstring—Optional leading icon — a registered FontAwesome icon name, not a Material ligature. Keep the label even with an icon.
disabledboolean—Disables the button; not clickable.
loadingboolean—Disables the button and shows a spinner in place of the label for async actions.

type'button' | 'submit'—Native button type attribute.

Checkbox · Radio · Switch — actual size

Selection controls capture choices: checkboxes for multi-select, radio buttons for one-of-many, and switches for instant on/off settings. Choosing the right one tells the operator how the choice behaves.

Checkbox = select none/some/all and apply on submit. Radio = pick exactly one from a small set. Switch = toggle a setting that takes effect immediately.

Best practices

Do

Slider

Continuous — actual size

Sliders let an operator choose a value, or a range, by dragging along a track — useful for soft, approximate settings like a geofence radius or a map's time window. When precision matters, prefer a number field.

Use a slider when the exact number is less important than the relative position, or when immediate visual feedback helps. For exact entry (e.g. an odometer), use a text field.

Variants

ContinuousAny value across the range (e.g. opacity, radius).
DiscreteSnaps to steps with tick marks (e.g. 5/10/15 km).
Range (two thumbs)Select a from–to span (e.g. a date/time window).

Best practices

Do
  • Show the current value.
  • Use discrete steps when only certain values are valid.
  • Pair with a field when exact entry is needed.
Don't
  • Don't use a slider for precise or wide-range numeric entry.
  • Don't hide the value.
  • Don't make the thumb hard to grab (keep a generous hit area).

Props

Page Header

The horizontal row at the top of almost every authenticated page — title (or a back-link) on the left, page-level actions on the right. Unlike most of this library, it isn't modeled on an idealized MD2 spec: it's a direct, faithful port of the real, currently-shipping fleet-web component, not an approximation.

Use PageHeader at the top of any page. Put PageHeader.Title (or a Breadcrumbs-based back-link) first and PageHeader.ButtonsContainer second — the real component has no other slots or props.

Best practices

Do
  • Put Title (or a back-link) in the left slot, ButtonsContainer in the right — that order, every time.
  • Build detail-page back-links from Breadcrumbs (back arrow + parent label), not a bare IconButton.
  • Keep it borderless, elevation-less, and non-sticky — that's what every real usage does.
Don't
  • Don't add a subtitle, icon, or backLink prop — the real component doesn't have one.
  • Don't put Tabs or a border-bottom inside PageHeader — that row belongs directly below it.
  • Don't make it sticky or elevated — no real usage does this.

  • Don't hard-code colours/sizes — use the theme tokens and these specs.
